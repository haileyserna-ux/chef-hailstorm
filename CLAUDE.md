## What this is

**Chef Hailstorm** is an AI-powered recipe co-pilot web app. Hailey uses it to plan a week of meals, scale servings, swap ingredients, log to MyFitnessPal, and save favorites to a recipe book.

## Architecture (one-paragraph version)

It's a **single static HTML file**. All CSS lives inline in one `<style>` block, all JS lives inline in one `<script>` block. No build system, no framework, no package.json. State persists in `localStorage`. The app calls the Anthropic Messages API directly from the browser using `anthropic-dangerous-direct-browser-access: true` and Hailey's API key (which she pastes into a setup overlay on first load).

## Key features currently implemented

- **This week** view — generates 4 weekly recipes (Mon–Thu) from Claude using pantry + preferences
- **Recipe cards** — clickable, open a dedicated recipe detail page in a new tab via `?recipe=ID` URL param
- **Recipe detail page** — full instructions, why-it's-healthy callout, sources (2–3 AI-suggested URLs from real recipe sites), servings adjuster (− / +) that scales ingredient quantities, per-ingredient swap (with AI-suggested alternatives + custom text input), Log to MFP button, Save to book button
- **Per-recipe swap** on weekly cards — 🔄 button with optional reason text field; calls Claude for one replacement recipe
- **Pantry** — track ingredients you already have
- **Shopping list** — auto-generated from weekly recipes minus pantry items, grouped into Proteins / Produce / Pantry & dry goods / Dairy
- **Recipe book** — save favorites
- **Log to MFP page** — sidebar nav item; each recipe gets a button that opens MyFitnessPal pre-searched

## Important code conventions / gotchas

1. **All top-level `const` AND `let` declarations sit at the top of the script, before init runs.** `COLORS`, `PILL_CLS`, `DAYS`, `SHOP_SECTIONS`, plus the recipe-page state vars (`currentRecipe`, `currentRecipeBaseServings`, `currentRecipeServings`, `originalIngredients`, `recipeIndexInWeekly`) MUST sit before the `// ══ INIT ══` block. If they're declared later, init code triggers a TDZ error ("Cannot access 'X' before initialization") and the page goes blank or shows an error. We've hit this twice — once with COLORS, once with currentRecipe. Functions are hoisted, so `function foo()` declarations can live anywhere; `let`/`const` cannot.
2. **Recipes need a stable `id`.** New recipes generated by `generateWeeklyRecipes` get `r_<base36 timestamp>_<random>` IDs. Old saved recipes without IDs get one generated on click. The `?recipe=ID` URL param relies on this.
3. **Recipe schema** (current shape — keep this in sync if you change the API prompt):
   ```js
   {
     id, emoji, name, description, kcal, protein, time, servings,
     tags: [...], cuisine,
     ingredients: [...],
     instructions: [...],
     whyHealthy: "...",
     sources: [{title, url}, ...]
   }
   ```
4. **Ingredient scaling regex** — `/^(\s*)((?:\d+\s+\d+\/\d+)|(?:\d+\/\d+)|(?:\d+(?:\.\d+)?))/` handles mixed fractions, fractions, and decimals. `formatQuantity` cleans output back into pretty fractions (1/4, 1/3, 1/2, 2/3, 3/4).
5. **Backward compat** — old saved recipes may lack `instructions` / `whyHealthy` / `sources` / `id`. Detail page handles missing fields gracefully (empty-state messages or skips the section).
6. **CSS theming** uses CSS variables: `--purple`, `--teal-light`, etc. Stick to existing palette unless redesigning.
7. **`body.recipe-page-mode`** class hides the sidebar on the detail page.
